# C++ 库加载不了原因排查

## 结论

当前工程里真正会被 ArkTS 加载的原生库是 `libentry.so`，但它现在并不是由你新增的 C++ 文件生成的，而是只由下面两个 C 文件编译出来：

- `entry/src/main/cpp/napi_init.c`
- `entry/src/main/cpp/tcp_client.c`

对应配置在：

- `entry/src/main/cpp/CMakeLists.txt`

当前内容是：

```cmake
add_library(entry SHARED napi_init.c tcp_client.c)
```

这意味着：

1. `data.cpp`、`IEC104.h`、`setting.h`、`quality.h` 这些文件虽然已经放进目录里，但根本没有参与 `libentry.so` 的编译。
2. 运行时加载到的还是旧的 C 版本 native 模块，不会包含你这些 C++ 实现。

## 第二个阻塞点

就算现在把 `data.cpp` 直接加进 `CMakeLists.txt`，也大概率会先卡在编译阶段，而不是顺利生成可加载的 C++ so。

原因是这些 C++ 文件带了当前工程没有提供的 Qt 依赖：

- `data.cpp` 引用了 `#include <QString>`
- `data.cpp` 引用了 `#include <QQueue>`
- `setting.h` 引用了 `#include <QString>`
- `quality.h` 引用了 `#include <QString>`

另外，`data.cpp` 还引用了一个当前仓库里不存在的头文件：

- `#include "def.h"`

仓库内没有搜到 `def.h`。

这说明这批 C++ 代码更像是从 Qt/桌面工程直接搬过来的，不是当前 OpenHarmony NAPI native 工程可直接编译的代码。

## 关于“加载不了”的准确说法

如果你说的“加载不了 C++ 库”是指 ArkTS 侧拿不到你新写的 C++ 能力，当前根因不是加载器先坏了，而是：

1. C++ 文件没有被编进 `libentry.so`
2. 这些 C++ 文件本身还依赖 Qt 和缺失头文件，暂时也不能直接编进来

## 当前工程里原生加载链路

ArkTS 侧目前导入的是：

```ts
import * as tcpClientModule from 'libentry.so';
```

原生模块注册名来自：

- `entry/src/main/cpp/napi_init.c`

注册方式是：

```c
NAPI_MODULE(entry, Init)
```

这套链路对应的是现有 C 版 TCP 模块，不是你新增的那批 C++ 业务代码。

## 如果要真正接入 C++，下一步应该做什么

1. 先决定这批代码是否真的要在 OpenHarmony native 层运行。
2. 如果要运行，先去掉 Qt 依赖，把 `QString`、`QQueue` 等替换成标准 C++ 或平台可用实现。
3. 补齐缺失的 `def.h`，或者把依赖内容合并进当前工程。
4. 再修改 `entry/src/main/cpp/CMakeLists.txt`，把需要的 `.cpp` 文件真正加入 `add_library(...)`。
5. 如果 C++ 侧要暴露给 ArkTS，还要把导出函数接进 `napi_init.cpp` 或 `napi_init.c` 的 NAPI 注册表。

## 补充

当前项目之前确实存在过“按 HarmonyOS 产品编译，导致 OpenHarmony 真机无法正确注册 native so”的问题，但从现在仓库里的配置看，这个老问题已经基本朝正确方向修过了：

- 根 `build-profile.json5` 默认产品已是 `OpenHarmony`
- `entry/build-profile.json5` 已保留 `runtimeOnly.packages: ["libentry.so"]`

所以这次更直接的问题，不是“OpenHarmony 不支持 C++ so”，而是“你的 C++ 代码还没有变成当前可加载的 so”。

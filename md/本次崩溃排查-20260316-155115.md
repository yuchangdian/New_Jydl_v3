# 本次崩溃排查 2026-03-16 15:51:15

## 崩溃日志

日志文件：

`E:\baidu\arkTS_code\log_error\jscrash-com.example.new_jydl_v3-19700101155115.169.txt`

核心报错：

- `Reason: TypeError`
- `Error message: Cannot read property add of undefined`
- `at onCreate (entry|entry|1.0.0|src/main/ets/entryability/EntryAbility.ts:121:80)`

## 直接原因

这次不是 TCP 线程崩了，也不是 `startTcpClient()` 调用时崩了。

直接崩溃点在 `EntryAbility.ets` 的 `onCreate()`：

```ts
hilog.info(DOMAIN, 'testTag', 'Test NAPI 2 + 3 = %{public}d', testNapi.add(2, 3));
```

当前文件里存在：

```ts
import testNapi from 'libentry.so';
```

但运行时 `testNapi` 是 `undefined`，因此调用 `testNapi.add(...)` 时直接触发：

`Cannot read property add of undefined`

## 为什么会是 undefined

当前 native 实现文件：

`E:\baidu\arkTS_code\New_Jydl_v3\entry\src\main\cpp\napi_init.c`

真实导出的只有 3 个方法：

- `startTcpClient`
- `stopTcpClient`
- `isTcpClientRunning`

并没有导出：

- `add`

而类型声明文件：

`E:\baidu\arkTS_code\New_Jydl_v3\entry\src\main\cpp\types\libentry\Index.d.ts`

却声明了：

```ts
export const add: (a: number, b: number) => number;
```

这和真实 native 导出不一致。

另外，`EntryAbility.ets` 使用的是：

```ts
import testNapi from 'libentry.so';
```

这是默认导入写法，但当前 `libentry.so` 在项目里实际按命名导出使用，并没有可靠的默认导出对象。

## 当前项目中的连带风险

除了 `EntryAbility.ets` 以外，`Index.ets` 里也还保留了：

```ts
import testNapi from 'libentry.so';
```

虽然当前页面里没有直接调用 `testNapi.add(...)`，但这类测试代码应当一并清理，否则后续仍可能再次触发同类问题。

## 结论

这次崩溃的主因是：

1. `EntryAbility.ets` 里保留了测试代码 `testNapi.add(2, 3)`
2. `libentry.so` 实际没有导出 `add`
3. 还使用了默认导入 `import testNapi from 'libentry.so'`

所以应用在 `onCreate()` 阶段先因为测试代码崩掉了，甚至还没进入真正的 TCP 客户端启动逻辑。

## 建议修改方向

最小修复：

1. 删除 `EntryAbility.ets` 中的：
   - `import testNapi from 'libentry.so';`
   - `testNapi.add(2, 3)` 调试日志
2. 删除 `Index.ets` 中无用的：
   - `import testNapi from 'libentry.so';`
3. 删除 `Index.d.ts` 中错误的：
   - `export const add: (a: number, b: number) => number;`

如果确实需要 `add` 这个测试方法，则必须反过来在 `napi_init.c` 中真实导出 `add`，并确保 ArkTS 侧按正确导出方式调用。

# data.h 改成 C++ 形式带注释指导

## 目标

这份文档只讲一件事：

**如何把 `entry/src/main/cpp/data.h` 从原来偏 Qt / C 风格的写法，整理成当前工程需要的标准 C++ 形式。**

这里的“改成 C++ 的形式”，这次主要指下面几类改动：

1. 头文件依赖改成标准 C++ 可用头文件
2. 队列声明从 `QQueue` 改成 `std::deque`
3. 队列长度宏改成更清晰的 C++ 侧命名
4. 其它业务结构体、宏、CRC 逻辑保持不变

这次不要扩散修改范围。  
`data.h` 的重点不是“重写业务结构”，而是：

**把它改成能和当前 `data.cpp` 的标准 C++ 实现一致。**

---

## 当前应该达到的目标状态

当前 `data.cpp` 已经按标准 C++ 使用了：

- `std::deque<...>`

所以 `data.h` 也必须同步成同样的声明形式，否则定义和声明会不一致。

你最终希望 `data.h` 达到的关键状态是：

1. 包含 `<cstdint>`
2. 包含 `<deque>`
3. 队列 extern 声明全部是 `std::deque<...>`
4. 队列长度宏名不再叫 `QQueue_Maxlength`

---

## 一、哪些地方要改

`data.h` 这次真正要改的地方很集中，主要只有 3 类：

### 1. 头文件 include 区

要明确引入标准 C++ 所需头文件：

- `<cstdint>`
- `<deque>`

### 2. 队列长度宏

把：

```cpp
#define QQueue_Maxlength 100
```

改成更符合当前标准 C++ 语义的名字。

### 3. 队列 extern 声明

把原来所有：

```cpp
extern QQueue<...> Xxx_Queue;
```

改成：

```cpp
extern std::deque<...> Xxx_Queue;
```

---

## 二、不要改的地方

这次容易误伤的地方也很明确。

下面这些都不要动：

1. 所有业务结构体定义
2. 所有业务宏值
3. `CRC32` / `CRC16` 逻辑
4. `#pragma pack(1)` 和 packed 结构体布局
5. 各种 `extern` 业务数据对象

原因很简单：

这次是“容器类型和头文件对齐”，不是业务协议重构。

---

## 三、具体改法

## 第 1 处：include 区

### 目标

让 `data.h` 自己就具备下面这些类型的定义来源：

1. `uint8_t` / `uint16_t` / `uint32_t`
2. `std::deque`

### 推荐写法

把文件头部整理成下面这种形式：

```cpp
#ifndef DATA_H
#define DATA_H

#include <iostream>
#include <complex.h>
#include <cstdint>  // 提供 uint8_t / uint16_t / uint32_t 这类固定宽度整数类型
#include <deque>    // 提供 std::deque，用来替代原来的 QQueue
```

### 为什么要这样改

#### `#include <cstdint>`

`data.h` 里大量用了：

- `uint8_t`
- `uint16_t`
- `uint32_t`

这些类型标准来源就是 `<cstdint>`。  
加上它以后，头文件本身依赖更完整，不需要再“碰巧依赖别的头间接带进来”。

#### `#include <deque>`

后面你要把所有队列声明改成 `std::deque<...>`，那这个头文件必须自己引入 `<deque>`。

### 注释要求

你要求“修改过的地方全部都要注释清楚并带讲解”，那 include 这两行就应该直接带注释：

```cpp
#include <cstdint>  // 提供固定宽度整数类型，避免 uint8_t / uint16_t / uint32_t 无定义
#include <deque>    // 提供标准 C++ 双端队列，用来替代原先的 QQueue
```

---

## 第 2 处：队列长度宏

### 原来可能是这种写法

```cpp
#define QQueue_Maxlength 100
```

### 建议改成

```cpp
#define DATA_QUEUE_MAX_LENGTH 100  // 标准 C++ 队列统一长度上限，替代旧的 QQueue_Maxlength 命名
```

### 为什么要改

旧名字里直接带着 `QQueue`，这是 Qt 容器时代的命名。  
既然你现在已经不用 Qt 容器了，这个名字继续保留会误导后续维护的人。

### 这里的边界

这一步只改“名字”和“注释表达”，不改值。

也就是说：

- `100` 这个值先保持不动

这样才能保证逻辑不变，只做容器命名和依赖迁移。

---

## 第 3 处：队列 extern 声明

这是 `data.h` 这次最关键的部分。

### 原来的 Qt 风格写法

```cpp
extern QQueue<YC_BaseFreq_Struct> BaseFreq_Queue;
extern QQueue<YC_Energy_Struct> Energy_Queue;
extern QQueue<YC_HarmonicU_Struct> HarmonicU_Queue;
extern QQueue<YC_HarmonicI_Struct> HarmonicI_Queue;
extern QQueue<RemoteSignal_Change_Struct> YX_Change_Queue;
extern QQueue<SOE_Node_Struct> SOE_Node_Queue;
```

### 现在标准 C++ 写法

```cpp
extern std::deque<YC_BaseFreq_Struct> BaseFreq_Queue;              // 基波遥测数据队列，改为标准 C++ 双端队列声明
extern std::deque<YC_Energy_Struct> Energy_Queue;                  // 电能数据队列，改为标准 C++ 双端队列声明
extern std::deque<YC_HarmonicU_Struct> HarmonicU_Queue;            // 电压谐波数据队列，改为标准 C++ 双端队列声明
extern std::deque<YC_HarmonicI_Struct> HarmonicI_Queue;            // 电流谐波数据队列，改为标准 C++ 双端队列声明
extern std::deque<RemoteSignal_Change_Struct> YX_Change_Queue;     // 遥信变化数据队列，改为标准 C++ 双端队列声明
extern std::deque<SOE_Node_Struct> SOE_Node_Queue;                 // SOE 事件节点队列，改为标准 C++ 双端队列声明
```

### 为什么选 `std::deque`

这里不建议写成 `std::queue`，原因是：

1. `std::queue` 是容器适配器，接口更窄
2. 后续如果要查看内容、按头尾操作、扩展维护，`std::deque` 更灵活
3. `data.cpp` 当前已经定义成了 `std::deque<...>`，声明必须和定义一致

### 注释为什么要写在每一行

因为你要求“修改过的地方全部都要注释清楚并且带讲解”。  
那这里最好的做法不是只在上面写一句总注释，而是每个声明都写清它是什么队列、为什么改。

这能解决两个问题：

1. 后面看头文件的人不需要回头翻 `data.cpp`
2. 一眼就能看出这是“Qt 容器迁移后的标准 C++ 容器声明”

## 四、推荐你在 `data.h` 里直接写成下面这样

下面这段只展示本次应该改动的部分，方便你直接照着改。

```cpp
#ifndef DATA_H
#define DATA_H

#include <iostream>
#include <complex.h>
#include <cstdint>  // 提供 uint8_t / uint16_t / uint32_t 等固定宽度整数类型
#include <deque>    // 提供 std::deque，作为 Qt 的 QQueue 替代容器

// ...前面的业务结构体、宏、extern 保持不动...

#define DATA_QUEUE_MAX_LENGTH 100  // 标准 C++ 队列统一长度上限，替代旧的 QQueue_Maxlength 命名

extern std::deque<YC_BaseFreq_Struct> BaseFreq_Queue;              // 基波遥测数据队列，改为标准 C++ 双端队列声明
extern std::deque<YC_Energy_Struct> Energy_Queue;                  // 电能数据队列，改为标准 C++ 双端队列声明
extern std::deque<YC_HarmonicU_Struct> HarmonicU_Queue;            // 电压谐波数据队列，改为标准 C++ 双端队列声明
extern std::deque<YC_HarmonicI_Struct> HarmonicI_Queue;            // 电流谐波数据队列，改为标准 C++ 双端队列声明
extern std::deque<RemoteSignal_Change_Struct> YX_Change_Queue;     // 遥信变化数据队列，改为标准 C++ 双端队列声明
extern std::deque<SOE_Node_Struct> SOE_Node_Queue;                 // SOE 事件节点队列，改为标准 C++ 双端队列声明

// ...后面的 extern / CRC / 结构体保持不动...

#endif // DATA_H
```

## 五、每个修改点的讲解总结

### 修改点 1：加 `<cstdint>`

#### 作用

让 `uint8_t / uint16_t / uint32_t` 的来源明确化。

#### 为什么重要

如果不显式包含，有时会依赖别的头文件“顺带包含”，这会让头文件变得不稳定。

---

### 修改点 2：加 `<deque>`

#### 作用

为 `std::deque` 声明提供标准库定义。

#### 为什么重要

`data.h` 既然自己声明了 `std::deque<...>`，那它自己就必须包含 `<deque>`。

---

### 修改点 3：宏名从 `QQueue_Maxlength` 改成 `DATA_QUEUE_MAX_LENGTH`

#### 作用

把旧 Qt 时代命名改成现在标准 C++ 时代的中性命名。

#### 为什么重要

名字是维护成本的一部分。  
你已经不再使用 `QQueue`，继续保留旧名字只会制造误导。

---

### 修改点 4：`QQueue<...>` 改成 `std::deque<...>`

#### 作用

让头文件声明和当前 `data.cpp` 的标准 C++ 定义保持一致。

#### 为什么重要

如果头文件还是 Qt 类型、实现文件已经是 C++ 类型，那会直接导致声明和定义不匹配。

---

## 六、改完后你应该检查什么

改完 `data.h` 后，建议你自己做下面几项检查：

### 1. 这个文件里不应再出现 `QQueue`

直接搜：

```text
QQueue
```

如果还有，说明还有漏改。

### 2. 这个文件里应当能看到 `<deque>`

因为队列声明已经改成了 `std::deque`。

### 3. 队列宏名不应再带 Qt 痕迹

建议确认：

```text
DATA_QUEUE_MAX_LENGTH
```

已经替代掉旧名字。

### 4. `data.cpp` 和 `data.h` 队列类型必须一致

也就是：

- `data.h` 里是 `std::deque`
- `data.cpp` 里定义也必须是 `std::deque`

这两个地方绝对不能一个是 `deque` 一个还是 `QQueue`。

---

## 七、这次为什么只改这些地方

因为这次任务是：

**把 `data.h` 改成标准 C++ 形式。**

这件事的边界不是“重写所有代码”，而是：

1. 改头文件依赖
2. 改容器声明
3. 改宏命名
4. 用注释把改动原因写清楚

如果你再继续去改结构体字段、CRC、业务枚举、packed 对齐，那就已经不是“data.h 去 Qt / 转 C++ 形式”，而是在做业务层面的重构了。

---

## 最后一句

`data.h` 这次最重要的不是“改得多”，而是“改得准”。

只要你把：

1. `<cstdint>`
2. `<deque>`
3. `DATA_QUEUE_MAX_LENGTH`
4. `std::deque<...>` 的 extern 声明

这几处改对，并且注释写清楚，`data.h` 这一轮就算改到位了。

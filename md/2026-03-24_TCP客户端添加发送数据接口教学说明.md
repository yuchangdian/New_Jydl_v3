## 你要新增的目标

你要补的是一整条“发送数据链路”：

1. 在 `tcp_client.c` 里增加原生发送函数
2. 在 `tcp_client.h` 里声明这个函数
3. 在 `napi_init.cpp` 里增加一个 NAPI 包装函数
4. 把这个新函数导出给 ArkTS
5. ArkTS 再去调用这个导出的发送接口

这 5 步缺一不可。

---

## 设计原则

这次新增接口，建议你坚持下面 4 条原则。

### 原则 1：不要在 ArkTS 里直接碰 socket

ArkTS 只负责把字符串或字节传下来。

真正的 `send(...)` 应该放在原生层，也就是：

- `tcp_client.c`

因为 socket 描述符 `g_socket_fd` 在这里管理，线程锁也在这里。

### 原则 2：发送函数必须加锁

你当前的连接状态、socket fd 都受：

- `g_state_mutex`

保护。

所以发送时也必须在同一把锁下取 `g_socket_fd`，否则会出现这类问题：

- 发送过程中连接刚好被关闭
- 线程正在重连时，发送拿到脏 fd
- stop 和 send 并发冲突

### 原则 3：先做“发字符串”版本

教学上我建议你先做：

- ArkTS 传入字符串
- 原生层按 UTF-8 字节发送

原因是它最容易打通全链路。

等你这个版本稳定后，再扩展：

- `ArrayBuffer`
- `Uint8Array`
- 二进制协议帧

### 原则 4：返回值先做成布尔值

第一版不要一上来做复杂错误码。

建议先返回：

- `true` 发送成功
- `false` 发送失败

这样你能先把调用链路跑通。

后面如果需要，再改成：

- 返回发送字节数
- 返回错误码
- 抛 NAPI 异常

## 第 1 步：在 `tcp_client.h` 增加函数声明

文件：

- [tcp_client.h](/E:/baidu/arkTS_code/New_Jydl_v3/entry/src/main/cpp/tcp_client.h)

你现在有这 3 个声明：

```c
int tcp_client_start(const char *host, int port);
void tcp_client_stop(void);
int tcp_client_is_running(void);
```

你需要新增第 4 个声明：

```c
int tcp_client_send(const char *data, int length);
```

## 为什么这样设计

这里不要只传 `const char *data`，最好同时传 `length`。

原因有两个：

### 原因 1：更稳

如果以后发的数据里包含 `\0`，只靠字符串结束符就不可靠了。

### 原因 2：以后更容易扩展成二进制发送

你现在先发文本没问题，但接口设计成“指针 + 长度”，后面升级成二进制协议时不用再改函数签名。

## 第 2 步：在 `tcp_client.c` 里实现发送函数

文件：

- [tcp_client.c](/E:/baidu/arkTS_code/New_Jydl_v3/entry/src/main/cpp/tcp_client.c)

你要新增一个函数，函数名与头文件保持一致：

```c
int tcp_client_send(const char *data, int length)
```

## 这个函数应该做什么

它至少要做下面几件事。

### 1. 检查参数

你先判断：

- `data == NULL`
- `length <= 0`

如果参数不合法，直接返回失败。

这里建议返回 `0`。

### 2. 加锁读取连接状态

你要先 `pthread_mutex_lock(&g_state_mutex)`，然后检查：

- 当前是否已连接
- `g_socket_fd >= 0`

如果没有可用连接：

- 解锁
- 返回失败

### 3. 调用 `send(...)`

最简单的第一版可以这样理解：

- 对当前 `g_socket_fd` 调 `send`
- 把传入的数据发出去

发送后要检查返回值：

- `< 0` 表示失败
- `>= 0` 表示内核实际写入了多少字节

### 4. 解锁

无论成功失败，都要记得解锁。

### 5. 返回结果

第一版建议这样处理：

- 如果 `send` 返回值等于 `length`，返回 `1`
- 否则返回 `0`

## 为什么第一版可以这么写

严格来说，TCP 的 `send(...)` 可能发生“部分发送”。

也就是说：

- 你要发 100 字节
- 这次只发出去了 40 字节

完整做法应该循环发送，直到全部发完。

但为了先跑通链路，第一版你可以先把逻辑做成：

- 发一次
- 没全发完就当失败

等你基础版本稳定后，再升级成“循环发完为止”的版本。

## 第 4 步：在 `napi_init.cpp` 新增 NAPI 包装函数

文件：

- [napi_init.cpp](/E:/baidu/arkTS_code/New_Jydl_v3/entry/src/main/cpp/napi_init.cpp)

你现在导出了：

- `startTcpClient`
- `stopTcpClient`
- `isTcpClientRunning`
- `add`

你要新增一个：

- `sendTcpData`

## 这个函数要做的事

NAPI 包装函数的职责不是自己发 socket。

它只负责两件事：

1. 从 ArkTS 拿到参数
2. 调用 `tcp_client_send(...)`

## 建议参数形式

第一版建议 ArkTS 传一个字符串：

```ts
sendTcpData("hello")
```

所以你的 NAPI 函数要：

### 1. 校验参数个数

至少要有 1 个参数。

### 2. 校验参数类型

这个参数应该是字符串。

### 3. 把 NAPI 字符串转成 UTF-8 `char *`

你当前已经有现成范例：

- `GetRequiredString(...)`

这个函数现在用于读取 `host`，你完全可以复用同一种思路来读取待发送内容。

### 4. 计算长度

得到字符串后，拿它的字节长度。

### 5. 调用原生发送函数

调用：

```cpp
tcp_client_send(buffer, length)
```

### 6. 把结果转成布尔值返回给 ArkTS

如果发送成功，返回 `true`。

如果失败，返回 `false`。

- C++ 层存在这个函数
- ArkTS 仍然调用不到

所以这一步一定要检查。

## 你在实现时最容易踩的坑

## 坑 1：连接还没建立就发送

`startTcpClient(...)` 只是启动线程，不等于 socket 已经立刻连上了。

所以你可能会遇到：

- 刚点启动
- 立刻调用发送
- 结果发送失败

这是合理现象。

因为连接线程可能还在：

- DNS 解析
- connect
- 重连等待

## 处理建议

第一版先接受这个行为。

也就是：

- 没连上就返回 `false`

后面如果你需要，可以再补：

- “已连接”状态接口
- 发送队列
- 未连接时缓存待发数据

---

## 坑 2：把 `send` 写在错误线程里

你当前连接线程一直在 `recv(...)` 阻塞读取。

这不代表发送也必须放到那个线程里。

对 TCP socket 来说：

- 一个线程收
- 另一个线程发

是完全常见的做法。

重点不是“必须同一线程”，重点是：

- 对 `g_socket_fd` 的访问必须受锁保护

---

## 坑 3：忘了处理字符串内存释放

你在 `napi_init.cpp` 里如果用 `malloc` 风格去拿字符串，就要记得在调用结束后释放。

否则会产生内存泄漏。

你现在 `StartTcpClient(...)` 的写法里已经有这个模式，可以照着做。

---

## 坑 4：把发送成功误判为“连接正常”

有时 `send(...)` 一次成功，不代表连接绝对没问题。

但在第一版里，你可以接受这个近似判断。

也就是说：

- `send` 成功 -> 本次发送成功
- 不要把它上升理解为“整个连接状态完全健康”

---

## 更稳的第二版应该怎么升级

等你第一版跑通后，再考虑这 3 个升级点。

### 升级 1：循环发送

把“部分发送”补完整：

- 未发完就继续发
- 直到全部发完或出错

### 升级 2：增加连接状态接口

现在的 `isTcpClientRunning()` 更接近“线程是否启动”。

它不完全等于：

- socket 当前已连接

如果你要做严谨一点，后面可以再补一个：

- `isTcpClientConnected()`

### 升级 3：支持二进制发送

后面如果你发 IEC104、私有规约、二进制帧，最好把接口扩成：

- ArkTS 传 `ArrayBuffer`
- 原生层按字节发送

---

## 你这次最推荐的最小实现方案

如果你问我“教学上最合适、最稳的第一版怎么做”，我建议你只做下面这个范围：

1. `tcp_client.h` 增加 `int tcp_client_send(const char *data, int length);`
2. `tcp_client.c` 实现这个函数
3. `napi_init.cpp` 增加 `sendTcpData`
4. ArkTS 传普通字符串
5. 返回布尔值

这样改动最小，最容易验证，也最不容易把现有接收逻辑弄乱。

## 一句话总结

你当前的 TCP 客户端确实没有发送接口。

正确添加方式不是只改一个文件，而是补完整条链路：

- `tcp_client.h` 声明发送函数
- `tcp_client.c` 实现 `send(...)`
- `napi_init.cpp` 包装成 NAPI
- `exports` 导出给 ArkTS

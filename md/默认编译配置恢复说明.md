# 默认编译配置恢复说明

## 1. 发生了什么

为了验证 `libentry.so` 在 OpenHarmony 5.1 设备上无法加载的根因，曾临时把项目根 [build-profile.json5](/E:/baidu/arkTS_code/New_Jydl_v3/build-profile.json5) 的默认产品从：

- `HarmonyOS`

切到了：

- `OpenHarmony`

这样做会触发 hvigor 去检查 OpenHarmony API 18 对应 SDK 的 license。

由于本机当前没有完成 OpenHarmony SDK license 接受，所以会直接报：

```text
The SDK license agreement is not accepted.
```

## 2. 当前已恢复

默认产品配置已经恢复为：

- `type: "HarmonyOS"`
- `targetSdkVersion: "6.0.1(21)"`
- `compatibleSdkVersion: "5.1.0(18)"`
- `runtimeOS: "HarmonyOS"`

也就是说，当前工程已经回到原来的可编译状态。

## 3. 验证结果

恢复后重新执行 `assembleHap`，结果为：

```text
BUILD SUCCESSFUL
```

说明这次“现在连编译都通不过”确实是因为默认产品被临时切成了 OpenHarmony，而不是 TCP 客户端代码把工程编坏了。

## 4. 后续原则

如果后面还要继续验证 OpenHarmony 真机上的 native 模块加载问题，不应该再直接改坏当前默认产品。

更安全的做法是二选一：

1. 单独加一个 OpenHarmony 产品配置
2. 等本机 OpenHarmony SDK license 处理好后，再单独切换验证

这样不会影响你当前的正常编译和发包。

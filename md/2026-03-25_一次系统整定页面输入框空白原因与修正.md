# 一次系统整定页面输入框空白原因与修正

## 现象

进入 `CommonSetting_PrimarySystem_Set.ets` 页面后，输入框全部显示为空白。

## 根因

页面打开时现在会优先读取 native：

- `CommonSetting_PrimarySystem_Set.ets` 的 `aboutToAppear()`

原来的逻辑是：

1. 调 `getPrimarySystemSetting()`
2. 只判断 `structValue.ready`
3. 只要 `ready = true`，就直接执行
   `createCommonSettingPrimarySystemDataFromStruct(structValue)`
4. 再整体替换 `formData`

问题出在：

- `CommonSettingPrimarySystemData.ets` 的 `formatPrimarySystemNumber()`

它遇到“不是有效 number”的值时会返回空字符串 `''`。

而之前的映射代码是直接这样赋值：

- `data.busPtPrimary = formatPrimarySystemNumber(...)`
- `data.busPtSecondary = formatPrimarySystemNumber(...)`
- ...

所以只要 native 返回对象里这些字段在 ArkTS 侧不是有效 number，
默认值就会被空字符串覆盖，最后整页输入框全空。

## 为什么会出现这种情况

因为当前页面对 native 数据的“可用性判断”太弱，只看了：

- `ready`

但没有继续判断：

- 各个数值字段是不是有效 number

于是会出现：

- native 结构体被认为“可读”
- 但 ArkTS 侧拿到的字段并不完整
- 映射时把默认值覆盖成空串

## 已做修正

做了两层保护：

### 1. 增加 struct 可用性判断

新增：

- `isUsableCommonSettingPrimarySystemStructValue(...)`

只有下面这些关键数值全部是有效 number 时，才允许用 native 数据覆盖界面：

- 母线 PT 一二次
- 母线零序 PT 一二次
- 线路 PT 一二次
- 保护 CT 一二次
- 零序 CT 一二次
- 测量 CT 一二次
- 线路额定容量

### 2. 映射时不再盲目覆盖默认值

现在改成：

- 先格式化
- 如果格式化结果非空，再写入 `CommonSettingPrimarySystemData`
- 否则保留默认值

这样即使 native 某个字段异常，也不会把整页输入框刷成空白。

- `assembleHap --analyze=normal --parallel --incremental`

结果：

- `BUILD SUCCESSFUL`

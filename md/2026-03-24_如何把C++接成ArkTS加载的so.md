# 如何把 C++ 接成 ArkTS 加载的那个 so

## 一句话结论

ArkTS 现在加载的不是“某个随便放进去的 C++ 文件”，而是一个 **NAPI 原生模块 so**。

在你这个工程里，ArkTS 当前加载的是：

- `libentry.so`

它对应的入口关系是：

1. `entry/src/main/ets/entryability/EntryAbility.ets` 里 `import * as tcpClientModule from 'libentry.so'`
2. `entry/src/main/cpp/CMakeLists.txt` 里把哪些源文件编译进 `libentry.so`
3. `entry/src/main/cpp/napi_init.c` 或 `entry/src/main/cpp/napi_init.cpp` 把 native 函数注册给 ArkTS

所以，想让你的 C++ 代码“变成被 ArkTS 加载的那个 so”，本质上不是把 `.cpp` 文件放进目录，而是要做到这几件事：

1. 让 C++ 代码能被当前 OpenHarmony native 工程编译
2. 让 C++ 源文件真正参与 `libentry.so` 的编译
3. 让 `libentry.so` 通过 NAPI 把函数导出给 ArkTS
4. 让类型声明、包配置、ArkTS 导入名保持一致

---

## 先说推荐方案

### 推荐方案：继续使用 `libentry.so`，把 C++ 代码编进去

这是最稳、改动最少的方案。

不要新搞一个完全独立的 `libxxx.so` 给 ArkTS 直接用，除非你确实要改模块名和整套导入链路。

你的当前工程已经把下面这些点接好了：

- `entry/build-profile.json5` 里有 `runtimeOnly.packages: ["libentry.so"]`
- `entry/oh-package.json5` 里依赖了 `libentry.so`
- `entry/src/main/cpp/types/libentry/oh-package.json5` 已经声明了 `libentry.so`
- ArkTS 已经在 `EntryAbility.ets` 里导入 `libentry.so`

所以最少改动路线就是：

**继续保留 `libentry.so` 这个名字，只把内部实现从“当前 C 文件”扩展成“C + C++ 文件共同生成的 so”。**

---

## 你现在卡住的真实原因

当前工程里真正参与编译 `libentry.so` 的只有：

```cmake
add_library(entry SHARED napi_init.c tcp_client.c)
```

也就是说，下面这些你新加的 C++ 文件目前没有参与 `libentry.so` 编译：

- `entry/src/main/cpp/data.cpp`
- `entry/src/main/cpp/data.h`
- `entry/src/main/cpp/IEC104.h`
- `entry/src/main/cpp/setting.h`
- `entry/src/main/cpp/quality.h`

而且这批 C++ 文件本身现在也不能直接编：

1. 依赖 Qt 类型
   - `QString`
   - `QQueue`
2. 引用了当前仓库里不存在的头文件
   - `def.h`
   - `CommonToolkit.h`

OpenHarmony NDK 这里不是 Qt 工程，所以这批代码必须先做“去 Qt 化”和“补依赖”处理，才能真正进 `libentry.so`。

---

## 改造目标图

建议你把结构改成下面这样：

```text
ArkTS
  -> import 'libentry.so'
  -> NAPI 桥接文件：napi_init.cpp
  -> 调用你自己的 C++ 业务代码：cpp_service.cpp / data.cpp / 其他 cpp
  -> 最终统一编译成 libentry.so
```

关键原则是：

**ArkTS 面对的始终只有一个可加载模块 `libentry.so`，你的 C++ 只是这个模块内部的实现。**

---

## 操作步骤

## 第 1 步：先决定你要哪种接入方式

你实际上有两种目标，改法不一样。

### 方式 A：ArkTS 不需要直接看到新 C++ 函数

例子：

- ArkTS 还是只调 `startTcpClient`
- 只是 `startTcpClient` 内部改成调用你的 C++ 实现

这种情况下：

- `Index.d.ts` 可以不新增太多导出
- ArkTS 代码基本不用改
- 只需要把 C++ 代码编进 `libentry.so`，再在 NAPI 现有函数里调用

### 方式 B：ArkTS 需要直接调用你新增的 C++ 能力

例子：

- ArkTS 想直接调用 `getSoeText`
- ArkTS 想直接调用 `loadRelaySetting`

这种情况下：

- 你要在 `napi_init.cpp` 里新增导出
- 要同步改 `Index.d.ts`
- 如果有 mock，也要同步改 `entry/src/mock/Libentry.mock.ets`

如果你现在只是想先把 C++ 真正编进 ArkTS 正在加载的 so，建议先走 **方式 A**，最容易跑通。

---

## 第 2 步：把 NAPI 桥接统一切到 C++

当前工程里同时存在：

- `entry/src/main/cpp/napi_init.c`
- `entry/src/main/cpp/napi_init.cpp`

但当前 `CMakeLists.txt` 只编了 `napi_init.c`。

如果你后面要接大量 C++ 代码，建议改成：

- 只保留 `napi_init.cpp` 作为 NAPI 桥接入口
- 不要同时把 `napi_init.c` 和 `napi_init.cpp` 都编进来

原因：

1. 你后面要调用 C++ 代码，用 `napi_init.cpp` 更自然
2. 两个文件里都在注册同一个模块名 `entry`
3. 如果两个都参与编译，容易出现重复注册或符号冲突

### 你要改的地方

把 `entry/src/main/cpp/CMakeLists.txt` 从现在的：

```cmake
add_library(entry SHARED napi_init.c tcp_client.c)
```

改成类似这样：

```cmake
cmake_minimum_required(VERSION 3.5.0)
project(New_Jydl_v3)

set(NATIVERENDER_ROOT_PATH ${CMAKE_CURRENT_SOURCE_DIR})
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

if(DEFINED PACKAGE_FIND_FILE)
    include(${PACKAGE_FIND_FILE})
endif()

include_directories(${NATIVERENDER_ROOT_PATH}
                    ${NATIVERENDER_ROOT_PATH}/include)

add_library(entry SHARED
    napi_init.cpp
    tcp_client.c
    data.cpp
)

target_link_libraries(entry PUBLIC libace_napi.z.so libhilog_ndk.z.so)
```

### 注意

这里的 `data.cpp` 只是示例。

你最后应该填入：

- 真正已经去 Qt 化完成、能编过的 `.cpp`
- 你希望被编进 `libentry.so` 的所有 native 业务实现文件

---

## 第 3 步：先把 C++ 文件改造成 OpenHarmony 能编的代码

这是最关键的一步。

### 3.1 去掉 Qt 依赖

你现在这批文件里有：

- `QString`
- `QQueue`

OpenHarmony 这里不是 Qt 工程，不能直接用。

建议替换关系：

1. `QString` -> `std::string`
2. `QQueue<T>` -> `std::deque<T>` 或 `std::queue<T>`

常见替换示例：

```cpp
#include <string>
#include <deque>
#include <sstream>
#include <cstdint>
```

例如：

```cpp
// 原来
QString Password = "123456";

// 建议改成
std::string Password = "123456";
```

```cpp
// 原来
QQueue<YC_BaseFreq_Struct> BaseFreq_Queue;

// 建议改成
std::deque<YC_BaseFreq_Struct> BaseFreqQueue;
```

### 3.2 字符串格式化也要改

你现在很多函数写成：

```cpp
QString SOE_StartString(uint16_t StartCode)
```

建议改成：

```cpp
std::string SOE_StartString(uint16_t startCode)
```

如果要拼接数字，建议用：

1. `std::ostringstream`
2. `std::to_string`

不要继续依赖 `QString::arg(...)`。

### 3.3 补齐缺失头文件

你现在还引用了仓库里没有的：

- `def.h`
- `CommonToolkit.h`

这两个问题必须先处理。

你有两个选择：

### 选择 1：把真实头文件补进工程

如果这两个头文件在你原来的 Qt 工程里确实存在，那就把它们拷到当前工程，并确保内容是当前 native 所需的最小子集。

### 选择 2：把依赖删掉，改成当前工程自己可维护的定义

如果只是少量宏、枚举、结构体，建议直接抽到当前工程自己的头文件，比如：

- `entry/src/main/cpp/native_defs.h`

这样后面更容易维护。

### 3.4 不要把 UI/桌面层逻辑硬搬进 native

如果原来的 Qt 代码里混入了这些能力，建议直接剥离掉：

1. Qt UI 类型
2. 文件对话框
3. QObject / signal / slot
4. 线程模型里依赖 Qt 事件循环的部分

ArkTS native 层只保留：

1. 算法
2. 协议解析
3. 数据结构
4. 可由 NAPI 包装的业务函数

---

## 第 4 步：给 C++ 业务代码一个清晰入口

不要让 `napi_init.cpp` 直接塞满业务逻辑。

建议新建一个中间层，例如：

- `entry/src/main/cpp/cpp_service.h`
- `entry/src/main/cpp/cpp_service.cpp`

### 推荐结构

`cpp_service.h`

```cpp
#ifndef CPP_SERVICE_H
#define CPP_SERVICE_H

#include <cstdint>
#include <string>

int32_t AddNative(int32_t a, int32_t b);
std::string GetSoeText(uint16_t code);

#endif
```

`cpp_service.cpp`

```cpp
#include "cpp_service.h"

int32_t AddNative(int32_t a, int32_t b)
{
    return a + b;
}

std::string GetSoeText(uint16_t code)
{
    switch (code) {
        case 1:
            return "测试文本";
        default:
            return "未知代码";
    }
}
```

以后你再把 `data.cpp`、`setting.h`、`IEC104.h` 的逻辑逐步沉到这个层里。

这样好处是：

1. NAPI 层干净
2. 业务层清楚
3. 后面排查编译问题更容易

---

## 第 5 步：在 `napi_init.cpp` 里把 C++ 能力导出给 ArkTS

这是“让 ArkTS 能真正调用 C++”的关键。

### 示例：导出一个 `getSoeText`

`napi_init.cpp` 里可以这样加：

```cpp
#include "napi/native_api.h"
#include "cpp_service.h"

#include <string>

static napi_value GetSoeTextWrap(napi_env env, napi_callback_info info)
{
    size_t argc = 1;
    napi_value args[1] = {nullptr};
    napi_get_cb_info(env, info, &argc, args, nullptr, nullptr);

    int32_t code = 0;
    if (argc < 1 || napi_get_value_int32(env, args[0], &code) != napi_ok) {
        napi_throw_type_error(env, nullptr, "code must be an integer");
        return nullptr;
    }

    std::string text = GetSoeText(static_cast<uint16_t>(code));

    napi_value result = nullptr;
    napi_create_string_utf8(env, text.c_str(), text.length(), &result);
    return result;
}
```

然后在导出表里注册：

```cpp
napi_property_descriptor desc[] = {
    {"startTcpClient", nullptr, StartTcpClient, nullptr, nullptr, nullptr, napi_default, nullptr},
    {"stopTcpClient", nullptr, StopTcpClient, nullptr, nullptr, nullptr, napi_default, nullptr},
    {"isTcpClientRunning", nullptr, IsTcpClientRunning, nullptr, nullptr, nullptr, napi_default, nullptr},
    {"add", nullptr, Add, nullptr, nullptr, nullptr, napi_default, nullptr},
    {"getSoeText", nullptr, GetSoeTextWrap, nullptr, nullptr, nullptr, napi_default, nullptr}
};
```

### 重点

ArkTS 能调用的不是 `QString`、不是 `std::string`、不是 C++ 对象本身。

ArkTS 能调用的是：

- 你在 `napi_init.cpp` 里包装过的 NAPI 导出函数

所以：

**C++ 业务代码只是内部实现，真正暴露给 ArkTS 的入口一定要经过 NAPI 包装。**

---

## 第 6 步：同步更新 ArkTS 类型声明

如果你新增了导出，就要改：

- `entry/src/main/cpp/types/libentry/Index.d.ts`

例如新增 `getSoeText` 后，类型文件要同步加：

```ts
export const getSoeText: (code: number) => string;
```

否则：

1. IDE 不认识这个导出
2. ArkTS 调用时类型提示不对
3. 容易出现“native 已导出，但声明没跟上”的错配

---

## 第 7 步：如果有 mock，也同步更新

当前工程里有：

- `entry/src/mock/Libentry.mock.ets`

如果你新增了 ArkTS 可见导出，mock 最好同步补上。

例如：

```ts
export const getSoeText = (code: number): string => {
  return `mock:${code}`;
};
```

这一步不是 native 编译必须，但对调试和测试友好。

---

## 第 8 步：保住下面这些“名字一致性”

你现在这个链路能成立，是因为这些名字是一套的。

### 8.1 so 包名

ArkTS 导入的是：

```ts
import * as tcpClientModule from 'libentry.so';
```

所以包名目前就是：

- `libentry.so`

### 8.2 NAPI 模块名

你当前 native 注册名是：

- `entry`

ArkTS 里也有：

```ts
requireNapi('entry')
```

### 8.3 类型包目录

当前类型目录是：

- `entry/src/main/cpp/types/libentry`

### 8.4 依赖声明

当前这些地方也已经跟 `libentry.so` 对齐：

1. `entry/oh-package.json5`
2. `entry/src/main/cpp/types/libentry/oh-package.json5`
3. `entry/build-profile.json5`

### 结论

如果你只是想让 C++ 变成 ArkTS 正在加载的那个 so：

**不要改 so 名字，不要改模块名，先继续沿用 `libentry.so` + `entry` 这套命名。**

这样改动最小。

---

## 第 9 步：编译后你应该验证什么

改完以后，不要只看有没有报错，要按下面顺序验证。

### 9.1 先确认构建产物时间变了

看下面路径里的 `libentry.so` 是否是新时间：

- `entry/build/default/intermediates/cmake/default/obj/arm64-v8a/libentry.so`

如果时间没变，说明你的 C++ 改动可能根本没进编译链。

### 9.2 如果你新增了导出，确认 ArkTS 端能拿到

例如在 `EntryAbility.ets` 里临时打印：

```ts
hilog.info(DOMAIN, 'testTag', 'typeof getSoeText = %{public}s', typeof tcpClientModule.getSoeText);
```

正常应该看到：

- `function`

### 9.3 如果只是内部实现替换，确认旧导出行为还正常

例如：

1. `startTcpClient`
2. `stopTcpClient`
3. `isTcpClientRunning`

这些旧接口还能调用，说明你没有把原本的模块加载链路弄断。

### 9.4 如果打包安装后仍然是旧逻辑

优先排查：

1. `CMakeLists.txt` 没把 `.cpp` 文件加进去
2. 改的是 `napi_init.cpp`，但实际编译的还是 `napi_init.c`
3. `libentry.so` 产物没更新
4. 安装到设备上的还是旧包

---

## 最小可跑通改法

如果你现在的目标只是“先证明 C++ 已经进了 ArkTS 正在加载的 so”，最小改法建议如下：

1. 不要一上来就迁 `data.cpp` 全量代码
2. 先新建一个最简单的 `cpp_service.cpp`
3. 在里面只写一个 `int32_t AddNative(int32_t a, int32_t b)`
4. 把 `napi_init.cpp` 接进 `CMakeLists.txt`
5. 在 `napi_init.cpp` 里把 `add` 改成调用 `AddNative`
6. 编译运行，确认 ArkTS 里的 `add(2, 3)` 结果正常

这一步跑通以后，才继续迁移：

1. `data.cpp`
2. `setting.h`
3. `IEC104.h`
4. `quality.h`

这样风险最低。

---

## 如果你坚持要改成一个新的 so 名字

例如你想让 ArkTS 直接导入：

```ts
import * as nativeModule from 'libjydlcpp.so';
```

那也可以，但你必须同步改完整条链路。

至少要改这些地方：

1. `entry/src/main/ets/entryability/EntryAbility.ets`
2. `entry/oh-package.json5`
3. `entry/src/main/cpp/types/libjydlcpp/oh-package.json5`
4. `entry/src/main/cpp/types/libjydlcpp/Index.d.ts`
5. `entry/build-profile.json5` 里的 `runtimeOnly.packages`
6. native 注册模块名
7. `requireNapi('xxx')` 的模块名

这条路线改动更大，所以当前不推荐。

---

## 最终建议

按优先级这样做：

1. 先别改 `libentry.so` 名字
2. 先把 `napi_init.c` 切成 `napi_init.cpp`
3. 先用一个最小的 `cpp_service.cpp` 验证 C++ 已经进 `libentry.so`
4. 再逐步迁移 `data.cpp` 这批代码
5. 迁移时先去 Qt 化，再补缺失头文件，再接 NAPI 导出

这样你就不会一口气把“构建问题、依赖问题、NAPI 导出问题、运行时加载问题”全混在一起。

---

## 你可以照着执行的顺序清单

1. 备份当前 `entry/src/main/cpp/CMakeLists.txt`
2. 把 `CMakeLists.txt` 改成编译 `napi_init.cpp`
3. 暂时不要再编译 `napi_init.c`
4. 新建一个最小 `cpp_service.h/.cpp`
5. 在 `napi_init.cpp` 里调用 `cpp_service.cpp`
6. 编译验证 `add()` 是否仍然可用
7. 通过后，再开始迁移 `data.cpp`
8. 把 `QString` 改成 `std::string`
9. 把 `QQueue` 改成 `std::deque` 或 `std::queue`
10. 处理 `def.h` 和 `CommonToolkit.h` 缺失问题
11. 如果新增 ArkTS 可见接口，再同步改 `Index.d.ts`
12. 必要时同步改 `Libentry.mock.ets`

---

## 补一句最关键的话

**ArkTS 不能直接加载“普通 C++ 代码”，只能加载“注册成 NAPI 模块并正确打包的 so”。**

所以你的目标不是“让 `data.cpp` 被加载”，而是：

**让 `data.cpp` 经过改造后，被编进 `libentry.so`，并通过 `napi_init.cpp` 暴露给 ArkTS。**

# 宽频谐波刷新问题半包修正说明

## 根因

宽频谐波页面之前即使已经加了：

- `500ms` 定时请求
- `500ms` 定时读取缓存

也仍然可能表现为“不刷新”。

原因不在页面定时器本身，而在 `tcp_client.cpp` 的接收缓冲逻辑：

- 每次 `recv` 都直接覆盖 `decodeBuffer_`
- 没有保留上一次未收完整的半包
- 谐波电压 / 电流报文长度较大，更容易被 TCP 拆成两次或多次 `recv`
- 一旦被拆包，原来的解码流程会把不完整报文直接丢掉

这样页面虽然一直在请求，native 侧却不一定能成功收到完整的谐波响应帧。

## 本次修正

这次把 `tcp_client.cpp` 的接收逻辑改成了：

### 1. 追加式接收

新收到的数据不再覆盖旧缓冲，而是追加到 `decodeBuffer_` 末尾。

### 2. 只解析完整帧

只有当缓冲区里已经满足：

- `commonAddr`
- `objectAddr`
- `dataLength`
- 完整 payload

时，才会进入解码函数。

### 3. 残包保留

如果最后一帧还没收完整：

- 本次不解码
- 剩余字节继续保留在缓冲区头部
- 等下次 `recv` 继续拼接

## 结果

这样像：

- `HarmonicU_Dsip`
- `HarmonicI_Dsip`

这类较大的谐波遥测包，就不会因为 TCP 半包被直接丢弃。

页面现在的刷新链路变成：

1. `500ms` 主动发送谐波请求
2. native 端可靠拼接完整响应帧
3. 更新 `HarmonicU_Dsip / HarmonicI_Dsip`
4. 页面再读取并刷新显示
